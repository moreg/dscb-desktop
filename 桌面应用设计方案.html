01

从 Web 全栈 → 桌面单机

原项目是 Python 后端 + React 前端 + SQLite 数据库的三层 Web 架构。桌面版用 Electron 主进程接管后端职责，数据库整体替换为本地文件。

原项目 ai-writer Web 全栈

Python 3.11 · FastAPI · SQLAlchemy · SQLite · Playwright · React 19
  • 后端跑 FastAPI 服务，前端走 HTTP API
  • 22 张 SQLite 表存全部数据
  • Playwright 抓番茄榜单 + 字体解密
  • LLM 在后端调用，Fernet 加密密钥
  • 需启动两个进程，依赖 Python 环境
→

桌面版 Electron 单机

Electron · Node.js · TypeScript · React 19 · 原生 fs
  • 主进程直接接管 LLM 调用 + 文件 I/O
  • 本地文件夹替代 SQLite（每项目一个目录）
  • 砍掉番茄榜单抓取、工作流、Agent
  • AES-256-GCM 加密 API Key
  • 双击即用，打包成 exe/dmg
02

整体架构：三层结构

渲染进程几乎直接复用原 React 前端，只把传输层从 HTTP 换成 IPC。原后端的业务逻辑（services）整体平移到主进程的 Repository 层——渲染进程零感知。

03

文件存储：一个项目 = 一个文件夹

核心设计理念：小说项目存成一个自包含、可独立携带的文件夹，可放任意本地路径（磁盘 / U 盘 / 网盘同步目录）。全局索引只存路径指针，不存正文。

📁

核心理念：项目可整体打包带走

复制一个文件夹 = 完整备份一部小说（含全部章节、大纲、记忆、伏笔）。可放进 Git 仓库版本管理，可用任意编辑器打开章节 Markdown，可丢进网盘同步。

04

SQLite 表 → 本地文件映射

核心 MVP 涉及 13 张表。列表型数据用 JSON items 数组，正文用 Markdown，审计日志用 JSONL 追加写。

SQLite 表落地位置格式
projects{project}/project.jsonJSON
chapterschapters/NNN.md + NNN.meta.jsonMarkdown + JSON
chapter_versionschapters/NNN.versions.jsonJSON
writing_stylesstyles/{name}.jsonJSON
charactersmemory/characters.jsonJSON
character_relationshipsmemory/relationships.jsonJSON
locationsmemory/locations.jsonJSON
foreshadowingsmemory/foreshadowings.json （保留 chain_id / parent_id 链）JSON
timeline_eventsmemory/timeline.jsonJSON
plot_pointsmemory/plot_points.jsonJSON
story_bibles / golden_fingers / worldviews合并为 memory/worldview.jsonJSON
outline_rules / generated_outlines / detailed_outlinesoutlines/*.jsonJSON
memory_historymemory/history.jsonlJSONL
generation_tasks不持久化 · 运行时内存—

所有 JSON 统一带 schemaVersion 与 updatedAt 字段。列表型用 items 数组；单例型（project/index）直接顶层平铺业务字段。

05

数据访问层：Repository + 原子写

每个领域一个 Repository，接口统一。写入采用「临时文件 + rename」原子操作，保证文件永不被写到一半——即使断电也不会损坏。

Repository 接口（统一）
interface Repository<T> {
  list(projectId): Promise<T[]>         // 列表
  get(projectId, id): Promise<T | null>  // 单条
  create(projectId, data): Promise<T>  // 新建
  update(projectId, id, data): Promise<T> // 更新
  delete(projectId, id): Promise<void>  // 删除
}

06

LLM 集成：主进程直连 + 密钥加密

provider 工厂从原 factory.py 移植到 TS。主进程直接 fetch 调用（无 CORS 问题），SSE 流式 token 经 IPC 推回渲染进程。

调用链

渲染进程invoke('llm:generate')
→
IPC 桥ipcMain.handle
→
主进程读 providers.enc 解密
→
LLM APIOpenAI / Claude / …
⤴
主进程SSE 解析
→
IPCwebContents.send
→
渲染进程逐 token 显示

API Key 加密存储

🔐 providers.enc

位置：<userData>/config/providers.enc
算法：AES-256-GCM（替代 Python Fernet）
替代原项目的 Fernet + .secret_key 方案。

🔑 密钥派生

机器特征（Win MachineGuid / mac IOPlatformUUID）+ 用户可选主密码，经 PBKDF2 派生。
支持从 ~/.claude/settings.json 等外部配置自动导入。

支持的 provider：OpenAI · Claude(Anthropic) · DeepSeek · MiniMax · 通义千问 · Codex。限流：Token Bucket（默认 0.1 req/s，burst=2）。

07

正文写作：AI 辅助章节生成

整个应用最核心的创作闭环。基于「细纲 + 记忆 + 风格」组装上下文，LLM 流式生成正文，支持去味润色与多版本对比。

上下文组装（Prompt 工程）

输入来源
本章细纲outlines/detailed.json — 情绪点 / 爽点 / 钩子 / 角色状态 / 剧情概要
出场人物档案memory/characters.json（按本章角色状态过滤）
伏笔任务memory/foreshadowings.json（pending 且本章需 plant / collect）

世界观约束memory/worldview.json（相关设定）

提示词模板从原 backend/app/prompts/ 移植到主进程，作为可编辑资源。

10

风险与对策

本地文件存储的主要风险集中在性能、并发冲突与密钥安全，均有明确对策。

风险对策
大量章节时目录扫描慢index.json 缓存章节列表，启动时增量校对
用户外部改文件导致不一致启动 / 聚焦时基于 mtime 增量重读 + 乐观锁
密钥丢失打不开主密码可重置（重置后重填 key），提供明文导出警告
跨平台路径 / 编码问题统一 path.join，文件名做安全化（去非法字符）
网盘同步冲突乐观锁 + 冲突文件保留两份并提示
✅

测试策略：覆盖率目标 80%+

单元（Repository 临时目录测试 / 加密 / provider mock fetch）· 集成（IPC 端到端，临时 userData）· E2E（Playwright 驱动 Electron，覆盖「新建项目→写章节→存文件→重开→数据还在」）。
