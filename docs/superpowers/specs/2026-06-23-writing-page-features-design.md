# 写作页面（ChapterEditor）缺失功能补全设计方案

本项目旨在补全写作页面中缺失的 12 项功能（涵盖 P0、P1、P2、P3 级需求），打通 AI 协同、日常写作辅助、人机联动等核心链路。

---

## 1. 架构与主要变化设计

### 1.1 状态管理与数据流动 (UI & State)
大部分新增功能基于纯前端逻辑实现，依赖组件的 State 及浏览器本地存储（`localStorage`）。
- **章节字数目标**：存储于 `localStorage`，键名 `ai-writer:word-target:${projectId}:${chapterNumber}`。
- **改写历史 diff 与规则回溯**：利用 `RewriteEntry` 已有的 `violationKey` 属性。如来自质检规则，在下拉菜单中通过 `report.violations` 或解析 `violationKey` 提取具体规则名称；如来自 AI 审稿建议，显示为“AI 改稿建议”（badge 文案沿用历史命名，实际数据源是流程面板折叠区）。
- **章节导航**：在组件加载时请求 `window.api.getChapterWordSummary(projectId)` 获取整本书的章节列表，以此判断当前章节索引，从而提供上一章/下一章导航。
- **查找替换 (Find & Replace)**：通过正则表达式以及 Range Selection API 对正文 `<textarea>` 进行匹配、高亮与替换。

### 1.2 主进程 IPC 接口扩展 (Backend & IPC)
为支持“AI 续写前临时补充上下文”这一功能，我们需要扩展续写指令。
- **IPC 消息**：修改 `write:generateChapter`，在其 payload 中支持 `tempContext?: string`（临时上下文）。
- **LLM 服务端**：在 `renderUserPrompt` 阶段，如果 `tempContext` 存在，则将其作为「临时写作要求」拼装入用户 Prompt 中。

## 2. 12个功能点的具体设计

### P0 级核心需求
#### ① AI 审稿建议「一键应用」与「应用全部」（位于续写流程面板内）
- **入口迁移**：原独立 `ReviewPanel` 侧栏（按钮 ✎ AI 改稿）已下线；卡片渲染与一键应用能力迁入**续写流程面板的"AI 审稿建议"折叠区**，避免重复 UI。
- **触发**：续写完成后自动调用 `reviewChapterStream`（`runPostGenerateReview`），结果灌入 `ChapterFlowPanel.props.reviewText`，折叠区立即渲染卡片。用户**不再手动触发**——只有续写后自动跑这一条入口。
- **单个应用**：折叠区每条卡片提供「应用」按钮。点击时：
  1. 通过 `computeSuggestionPositions` 预分配的位置找到建议 `quote` 在 draft 中的下标。
  2. 校验通过 `isRewritable` 后，调用 `props.onApplyRewrite(quote, candidate, buildReviewKey(index, pos))` 经 `pushRewrite` 压入历史栈，触发重跑质检。
- **应用全部**：折叠区顶部提供「应用全部」按钮，按位置倒序应用所有可自动替换的改写建议（避免下标漂移），逐条 `pushRewrite`。
- **定位跳转**：点击建议卡片时，通过新增的 `ChapterFlowPanel.onFocusQuote` 回调，让 `ChapterEditor` 在 `<textarea>` 中选中匹配的 `quote` 并自动聚焦/滚动定位。
- **复制说明**：无法自动应用的建议（缺 quote / rewrite 不合规），显示「复制说明」按钮，把 `rewrite + why` 写到剪贴板供用户手动粘贴调整。

#### ② 查找 & 替换 (Find & Replace)
- **UI 呈现**：在编辑器区顶部（输入区正上方）增加悬浮/嵌入式查找栏（`FindReplaceBar`），可通过 `Ctrl+F` 快捷键或工具栏上的「🔍 查找」按钮唤起。
- **功能特性**：
  - 包含「查找输入框」、「替换输入框」、「查找匹配计数（例如：`1 / 3`）」；
  - 包含「上一个」、「下一个」、「替换」、「全部替换」和「关闭」按钮；
  - 使用 React Ref 获取 `<textarea>` DOM，使用 `setSelectionRange()` 聚焦匹配文本。

### P1 级体验优化
#### ③ 上一章 / 下一章导航
- **UI 呈现**：在顶部标题栏「第 X 章」旁，添加「上一章」、「下一章」和「返回列表」按钮组。
- **逻辑实现**：通过 `getChapterWordSummary` 获取当前项目所有章节。如果存在上一章/下一章，则按钮可用，点击后触发 `onNavigateChapter(targetChapterNumber)`，返回 `App.tsx` 并切换路由状态。

#### ④ 番茄钟结束通知（声音 / 系统通知）
- **系统通知**：番茄钟归零时，检测并请求 HTML5 `Notification` 权限。若允许，弹出系统通知提示专注/休息结束。
- **通知音效**：使用 Web Audio API 在浏览器端纯代码合成自然、清脆的“双音符 Chime”提示音，避免因加载本地 `.mp3` 资源文件路径失败导致无声。

#### ⑤ 联动预览点击 → 改为内联卡片 (Replace window.alert)
- **UI 呈现**：点击预览高亮的词汇时，不弹出阻塞式的 `window.alert`，而是在鼠标点击位置弹出一个固定定位 (Float) 的内联详情卡片。
- **卡片内容**：
  - 人物：姓名、身份、性格，以及跳转至人物库的「查看档案」链接。
  - 伏笔：内容、状态。
  - 地点：地名、分类。
- **交互**：点击卡片外的任意位置或点击右上角“✕”关闭卡片。

---

### P2 级精细写作
#### ⑥ 章节字数目标
- **UI 呈现**：在左侧边栏「本次会话」字数统计下方，增加「本章目标」进度条。
- **交互**：默认显示“本章进度: X / Y 字”。点击“Y”可直接原位切换为数字输入框，修改后按回车或失去焦点即自动保存至 `localStorage`。

#### ⑦ AI续写前临时补充上下文
- **UI 呈现**：点击「✦ 续写」时，不再直接请求生成，而是弹出轻量级对话框（可勾选“以后不再提示，直接续写”）。
- **逻辑**：用户可输入临时指导建议（如“林晨在打斗中落入下风，但发现了对方的破绽”）。点击「确定」后，将建议作为 `tempContext` 传入续写流接口。

#### ⑧ 改写历史 diff 视图与触发规则标记
- **UI 呈现**：点击「↶ 撤销」下拉菜单展开改写历史列表时：
  - 每个历史记录卡片顶部显示“触发规则/来源”（badge 文案如“AI 改稿建议”、“禁用词: xx”、“规则 4: 空洞结尾”等——`AI 改稿建议` 字面保留，含义现在是"流程面板 AI 审稿建议折叠区应用的改写”），
  - 卡片中以 Diff 形式显示修改前后对照（`- 原文片段` 与 `+ 改写文本`），方便用户精细核对后再决定回退。

---

### P3 级辅助功能
#### ⑨ Markdown 排版预览模式
- **UI 呈现**：在编辑器底部的预览区，新增「联动高亮」与「Markdown排版」页签切换。
- **Markdown 排版逻辑**：简易解析常见 Markdown 语法（`#` 标题、`**` 加粗、`---` 分割线、换行段落），按小说排版样式（如首行缩进、行高 2.0）渲染为精美 HTML 视图。

#### ⑩ 全文统计 — 阅读时间估算
- **逻辑**：在「章节分析」面板字数统计旁，增加「预计阅读时间」单元格，按大众小说平均阅读速度（400字/分钟）向上取整估算。

#### ⑪ syncAllTrigger 传参修复
- **修复逻辑**：将 ChapterEditor 中漏传的 `syncAllTrigger` props 补上，连接到编辑器状态。当 ✍ AI 续写完成后，自动更新该 Trigger，直接跑完四大同步（细纲、记忆、节奏、图解），提升全自动流程体验。

## 3. 设计自评与验证方案

### 3.1 自动化测试
- 针对 `rewrite-history.ts` 补充 diff 信息和 `violationKey` 匹配回溯测试用例；
- 针对 LLM 提示词拼装逻辑，补充 `tempContext` 拼接验证单测。

### 3.2 手动验证流程
1. **P0 验证**：生成改稿建议，尝试一键应用和定位；使用 `Ctrl+F` 进行查找并批量替换。
2. **P1 验证**：点击上一章/下一章，观察状态是否无缝重载；等待番茄钟归零，验证提示音与系统通知。
3. **P2/P3 验证**：修改章节字数目标并重载，验证持久化；在续写前输入临时上下文，观察生成内容是否包含此方向；检查改写历史的 diff 样式与阅读时间估算。
